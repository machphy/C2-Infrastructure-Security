C2 Infrastructure Security – Career & Project Notes
==================================================

This document consolidates the career-facing value of the
"C2 Infrastructure Security: Detection & Defense Framework" project.

--------------------------------------------------
1. AIM OF THE PROJECT
--------------------------------------------------

The primary aim of this project is to design and implement a
blue-team focused detection framework that identifies
Command-and-Control (C2) activity using network behavior rather than
malware signatures or payload inspection.

The project specifically aims to:
- Detect periodic C2 beaconing over encrypted channels
- Reduce false positives using adaptive, host-based baselining
- Translate detection logic into SOC and SIEM operational artifacts
- Align detections with MITRE ATT&CK Command and Control techniques

--------------------------------------------------
2. MOTIVATION / WHY THIS PROJECT IS NEEDED
--------------------------------------------------

Modern malware and APT frameworks heavily rely on encrypted
communication (HTTPS, TLS) and legitimate protocols, making
signature-based and payload-based detection increasingly ineffective.

In real SOC environments:
- Payload inspection is often impossible due to encryption
- IOC-only detection lacks context and quickly becomes outdated
- Alert fatigue is caused by static rules and poor baselining

This project was motivated by the need to:
- Detect threats using metadata and behavioral patterns
- Demonstrate detection engineering rather than tool-only usage
- Bridge the gap between academic labs and real SOC operations

--------------------------------------------------
3. PROBLEM STATEMENT
--------------------------------------------------

How can Command-and-Control activity be detected reliably when:
- Network traffic is encrypted
- Malware payloads are not visible
- Destination infrastructure may change
- Legitimate applications generate high traffic volume

5.***************************
--------------------------------------------------

Q1: What is the core idea of your C2 project?

Answer:
The core idea is to detect command-and-control behavior not by
inspecting payloads or signatures, but by analyzing the timing and
structure of outbound connections. Using flow-level data, the project
identifies periodic beaconing from internal hosts to external
infrastructure and translates that behavior into SIEM and IOC-based
detections.

--------------------------------------------------

Q2: How does your beacon detector work?

Answer:
The detector processes network flow logs containing timestamps,
source IPs, destination IPs, and ports. For each source-destination
pair, it computes inter-arrival intervals and calculates their average
and standard deviation. Flows that exhibit highly regular intervals
with very low variance are flagged as potential C2 beaconing activity.
Because it relies only on timing metadata, it remains effective even
when traffic is encrypted.

--------------------------------------------------

Q3: What is the difference between the basic and adaptive detector?

Answer:
The basic detector uses a static threshold on interval variance to
identify beaconing. The adaptive detector first builds a per-host
baseline of normal connection behavior and then evaluates individual
flows relative to that baseline. This allows the system to suppress
false positives when regular behavior is normal for a host, and flag
only flows that are statistically anomalous for that specific host.

--------------------------------------------------

Q4: How does this project relate to SOC operations?

Answer:
The detection outputs are converted into IOC lists and SIEM-style
correlation rules that can be deployed in a SOC. Alerts are mapped
to MITRE ATT&CK Command and Control techniques, enabling structured
analysis, investigation, and escalation in a real-world SOC workflow.
